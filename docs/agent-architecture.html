1.5 Conversation Entry Points v0.3
Path A: New Conversation
User creates new conversation + selects Agent
-> Cold Assembly runs (full pipeline)
-> AgentRuntime created & cached
-> Conversation record inserted to DB
-> Ready for first message
Path B: Resume Conversation v0.3
User opens existing conversation from sidebar
-> Load conversation.agent_id from DB
-> Cold Assembly runs (same pipeline, same result)
-> Load message history from DB
-> ContextCompressor check: if history exceeds
  token budget, compress before first new message
-> AgentRuntime cached, ready for next message

Previous turn's exit_status (complete / cancelled / error)
does NOT affect resume. User simply sends next message.
Partial/cancelled messages are in history as-is.
Both paths converge at the same point: AgentRuntime cached + conversation in DB. From here, every user message follows the identical Hot Injection -> Engine flow. No special "resume" logic in the engine.
Both paths -> AgentRuntime ready

3.5 Confirm-Level Tool Approval Flow v0.3 DETAILED
Engine (Main Process)
4a. Tool Executor receives tool_call
ToolRegistry.get(name) -> check permission level
|
4b. Permission = 'confirm'
Create a Promise + resolver pair, store by callId
|
4c. emit tool:confirm { toolName, args, callId }
UI shows approval dialog with tool name + args preview
| await promise (blocks this tool_call)
4d-YES. Promise resolved (approved = true)
-> execute tool normally -> emit tool:result
4d-NO. Promise resolved (approved = false)
-> return ToolResult { isError: true,
  error: 'User denied: [tool_name] with args [summary]' }
-> model sees denial, adjusts approach
4d-CANCEL. AbortController.abort() triggered
-> Promise rejected -> tool skipped -> turn ends
Renderer (UI)
Receives tool:confirm event
Shows inline approval card:
|
Approval Card UI:
Tool: WriteFile
Path: src/utils/helper.ts
Preview: [diff / content snippet]

[ Approve ]  [ Deny ]  [ Always Allow ]
|
User clicks button
-> emit user:confirm { callId, approved }
Approve -> approved: true
Deny -> approved: false
Always Allow -> approved: true
  + update tool permission to 'auto' for this session
Parallel Confirm Handling
If model returns multiple tool_calls and some are confirm-level:

1. Auto tools execute immediately in parallel
2. Confirm tools each emit their own tool:confirm
3. UI shows multiple approval cards simultaneously
4. Each resolved independently via callId matching
5. Promise.allSettled() waits for all (auto + confirm) to resolve
6. All results appended to messages together
Timeout on Confirm Wait
No timeout on user confirmation.
Engine waits indefinitely (this is user's decision).

Only exits on:
- User approves/denies (normal path)
- User cancels the entire turn (AbortController)
- App closes (turn not persisted, treated as if never happened)

This is the same UX as Claude Code's permission prompt --
the system waits for human decision, no auto-timeout.

Token Usage Tracking v0.3
Collection Point
Every streamText() call returns usage object:
  { promptTokens, completionTokens, totalTokens }

One turn may have N model calls (N-1 tool loops + 1 final).
Engine accumulates per-turn:

turnUsage = {
  prompt_tokens: sum of all calls,
  completion_tokens: sum of all calls,
  model_calls: N,
  tool_calls: M,
  provider_id, model_id,
  conversation_id, created_at
}
Storage & Display
DB: token_usage table (one row per turn)
Written in same batch transaction as messages.

Event: usage included in turn:complete/cancelled/error
Renderer can show per-turn token count inline.

MVP Display: minimal
  - Per-turn: small token count badge on message
  - Per-conversation: total in conversation header
  - Settings: aggregated usage dashboard (later)

No cost calculation in MVP (requires price config
per model). Just raw token counts. Cost can be
added later as a computed view.
Context Overflow Guard (in-loop) v0.3
Detection
After step 4 (tool results appended), before looping back to step 1:

currentTokens = estimateTokens(allMessages)
threshold = agentRuntime.contextLimit * 0.9

if (currentTokens > threshold) {
  // Force graceful exit
}
Token estimation: rough char/4 heuristic is enough here.
Doesn't need to be exact -- the 90% threshold leaves
~10% buffer for the final model call.
Graceful Exit
When overflow detected, don't just abort -- do one final model call:

1. Append a system injection to messages:
  "Context limit approaching. Summarize your progress
  and what remains to be done. Do NOT call any more tools."

2. Call streamText() one last time (no tools registered)
3. Model outputs a summary for the user
4. Normal turn finalize (persist + emit turn:complete)

User sees a clear explanation of what was done
and what's left, can continue in next message
(which will trigger ContextCompressor on the
accumulated history).

Error Recovery Strategy
Tool Execution Error
Strategy: Model self-heals

Tool throws / times out
-> wrap as ToolResult { isError: true }
-> append to messages normally
-> model decides: retry / alternative / inform user

No framework-level retry for tools.
Model is the decision maker.
Model API Error
Strategy: Auto retry + fail fast

Network error / 5xx / rate limit
-> retry up to 3x, exp backoff (1s/2s/4s)
-> 429: respect Retry-After header
-> all retries failed:
  emit turn:error
  persist partial state + usage
  user sees error, can retry manually

No auto fallback to different model.
Tool Timeout
Strategy: Per-tool timeout + kill

Default timeouts:
  ReadFile: 5s
  WriteFile: 5s
  BashExec: 30s (configurable)
  Search: 10s

Timeout -> kill process
-> ToolResult { isError: true }
-> model adjusts approach
Message Persistence Strategy
Turn-Level Batch Write
During engine loop: all messages stay in memory

On turn end (any exit path):
  - text complete -> batch write
  - user cancel -> batch write (partial)
  - context overflow -> batch write (with summary)
  - max_iterations -> batch write
  - unrecoverable error -> batch write (partial)

Single DB transaction:
  INSERT user_message
  INSERT assistant_message (may be partial)
  INSERT tool_call records (0..N)
  INSERT tool_result records (0..N)
  INSERT token_usage record v0.3
  UPDATE conversation.updated_at
Why Turn-Level Batch
Simpler: one transaction, no partial state bugs

Faster: SQLite single transaction >> many small writes

Consistent: turn is either fully saved or saved as
partial with exit_status flag

Acceptable risk: worst case, app crashes mid-turn,
lose one turn. Previous turns are safe.

Same tradeoff as Claude Code, Cursor, etc.

Data Flow Summary
Entry: New conv | Resume conv --> Cold Assembly --> AgentRuntime (cached)
Hot: AgentRuntime + MemoryInjector(userMsg) --> PromptAssembler --> TurnContext --> AgentEngine
Loop: ContextBuild --> streamText() --> Router --> ToolExec (auto | confirm->await | deny) --> overflow? --> Loop
Exit: text | cancel | overflow | max_iter | error --> batch write (msgs + usage) --> emit event
Post: Conv end --[async]--> MemoryExtractor --> DB:memories
